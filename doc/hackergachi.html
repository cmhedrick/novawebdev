SVG Caveats & Customizations

Browser support: Riddle me this, Batman. What's the ONE thing you'd expect a browser to implement for SVG files? I'll give you a hint: It starts with an "S". Guess the ONE thing Chrome and Opera cannot do correctly? That's right: Scale. Or, rather, animate scaling. Apparently, although I have not verified it yet, they can do "transitions" which are non-repeating animations — open a page and any transitions take place from a starting state to an ending state and then stop. Also, in order to make the non-scaling animations work, one still has to add a lot of -webkit- prefixing nonsense to the CSS for non-Firefox browsers. Ug-ly!

First we figured out how to get a raster image converted to a vector image with Inkscape. Shift-Alt-B is your friend here.

Since we have multiple copies of the same SVG, in order to maintain strict HTML5 compliance, and save ourselves other headaches, duplicate id attributes need to be removed or made unique. For our purposes, in this case, most paths and group id attributes are unnecessary. So, remove all «id="path.*"». This eliminates the lion's share.

Next, go after «id="svg2"», «id="defs4"», and «id="metadata7"» attributes.

Finally, the last part, which is tricky: In this case there's a master group (<g>) that surrounds the actual drawing. That has to be made unique, as we need it to reference the specific copy. However, all other <g> tags should have the id attribute removed.

Rotation is the tricky one. SVG animation will rotate on all three axes. The Z axis appears to be the default. (Rotating around X causes looks like the image becomes thinned to a horizontal line, and Y to a vertical line, which makes sense. The tricky part is specifying the origin point of the rotation. To rotate Z around the center of the image, find the X and Y center of the image.

NOTE: The W3C HTML5 validator is not particularly fond of the custom XML Namespace lines:

       xmlns:dc="http://purl.org/dc/elements/1.1/"
       xmlns:cc="http://creativecommons.org/ns#"
       xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
       xmlns:svg="http://www.w3.org/2000/svg"

In fact, it has problems with the <metadata> section in general. The main purpose of this section appears to be to use the Resource Description Framework and Dublin Core (RDF and DC) specs to implement Creative Commons (CC) bibliographical / licensing stuff for web tools smart enough to catalog using it. (The curious can go look at http://www.w3schools.com/webservices/ws_rdf_intro.asp.) Adding such information should be possible by other means, particularly if the SVG is embedded in an HTML file as this one (times 4) is. And, since <defs /> is empty, it can go as well. But for reference, the naughty bits removed look like:

       <defs />
       <metadata>
         <rdf:RDF>
           <cc:Work
              rdf:about="">
             <dc:format>image/svg+xml</dc:format>
             <dc:type
                rdf:resource="http://purl.org/dc/dcmitype/StillImage" />
             <dc:title></dc:title>
           </cc:Work>
         </rdf:RDF>
       </metadata>
